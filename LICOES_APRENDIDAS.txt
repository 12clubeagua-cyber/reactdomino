# LICOES APRENDIDAS - PROTOCOLO DE PREVENCAO DE BUGS (REACTDOMINO)

Este arquivo e uma base de conhecimento obrigatoria para evitar a reincidencia de bugs tecnicos. 
Sempre que um bug for corrigido, este arquivo DEVE ser atualizado com a causa raiz e a solucao.

## 1. GEOMETRIA E POSICIONAMENTO (logic.js)
- **REGRA DE 6 (LINECOUNT):** A peca inicial (0,0) deve ser contabilizada no `lineCount` de ambos os lados (Side 0 e Side 1). Se for inicializada com 0, a primeira reta tera 7 pecas em vez de 6.
- **DIMENSOES EM CURVA (L-SHAPE):** Nunca use `wasDouble` como unico indicador de dimensao para curvas. Salve a orientacao real (`lastIsV`) da peca anterior. Buchas em curvas mudam de orientacao (ficam transversais ao fluxo antigo), o que altera qual eixo usa `TILE_W` ou `TILE_L`.
- **GAP DE SEGURANCA:** Mantenha sempre +2px em retas e +4px em projeções de curva para compensar bordas (1px cada lado) e evitar que o motor de colisao detecte sobreposicao.

## 2. INTERFACE E UX (renderer.js / ui.js / flowui.js)
- **DISPONIBILIDADE DE MENUS:** Botoes de configuracao (como Audio, Acessibilidade) devem ter seus containers de opcoes desacoplados do estado da partida (`STATE.isStarted`). Se o menu depende do DOM gerado no inicio do jogo, ele falhara no Lobby.
- **SOBREPOSICAO DE Z-INDEX:** Menus globais (como o painel de Audio) devem ter um `z-index` superior a tela de Lobby/Splash (`#start-screen`), que e 9999. Use 10001+ para garantir visibilidade fora da partida.
- **CACHE DO RENDERIZADOR (MAOS):** A otimizacao de renderizacao de maos (`drawHands`) deve incluir o conteudo das pecas locais no hash de estado (`JSON.stringify`). Se apenas `handSize` for verificado, a interface nao atualizara quando o Host enviar as pecas reais para o Cliente no inicio da rodada (pois o `handSize` ja era 7).
- **POSICIONAMENTO DE UI:** Botoes de controle (Menu, Emotes) nunca devem usar `position: fixed` no topo se houver um `header` de placar. Use `position: absolute` dentro do conteiner de jogo e posicione-os abaixo do header para evitar sobreposicao em telas mobile.

## 3. MULTIPLAYER E REDE (network.js / multiplayer.js)
- **BLIND HANDS:** Nunca inclua o objeto `window.STATE.hands` no broadcast global. Filtre para enviar apenas a mao do jogador local (`myHand`) e o tamanho das maos dos outros (`handSize`).
- **AUTORIDADE DO HOST:** O cliente nunca deve processar logica de posicionamento. Ele apenas recebe as coordenadas calculadas pelo Host para garantir sincronia visual total.

## 4. AMBIENTE DE TESTES (tests/*.js)
- **MOCKS DE DOM:** Sempre inclua `setAttribute`, `classList`, `appendChild` e `getElementById` nos mocks de `createElement` e do `document` global para evitar `TypeError` durante auditorias de UI em Node.js.
- **REFERENCIA WINDOW:** Use `window.xxx` em vez de `global.xxx` nos testes para manter paridade com o ambiente real do browser.

## 5. ACESSIBILIDADE E UX (accessibilitymanager.js / flowui.js / game.js)
- **ANUNCIOS DE FIM DE RODADA:** O motor de acessibilidade deve anunciar explicitamente o resultado final, placar e motivo do encerramento no `FlowUI.endRound`. Apenas feedback visual e insuficiente para inclusao.
- **FOCO AUTOMATICO (KEYBOARD NAV):** Ao iniciar o turno do jogador local, o sistema deve focar automaticamente na primeira peca jogavel (`.playable[tabindex="0"]`). Isso reduz drasticamente o numero de tabs necessarios para jogar via teclado.
- **REFORCO DE FOCO VISIVEL:** Use `:focus-visible` com `outline-offset` e `box-shadow` (glow) para garantir que o cursor de selecao seja visivel em todos os temas, especialmente para usuarios com baixa visao.

## 6. REGRAS DE OURO (GEMINI.md)
- **ASCII APENAS:** Proibido caracteres acentuados.
- **SEM IMPORTS:** Use apenas o objeto global `window`.
- **DIRETRIZ DE AUTONOMIA:** Identificou o erro? Corrija, teste, documente no LICOES_APRENDIDAS.txt e de push.

## 7. PERFORMANCE E OTIMIZACAO (renderer.js)
- **PARTICLE POOLING:** Implementado sistema de reaproveitamento de elementos DOM para efeitos visuais (`spawnEffect`). Isso reduz a carga no coletor de lixo e mantem a fluidez em dispositivos mobile.
- **INCREMENTAL RENDERING:** O tabuleiro so renderiza as novas pecas (`_renderedTilesCount`), preservando o que ja estava no DOM.

## 8. ACESSIBILIDADE AVANCADA (renderer.js / ui.js)
- **ARIA-LIVE DINAMICO:** O sistema de anuncios agora cria um elemento `aria-live="polite"` automaticamente se o gestor de acessibilidade estiver ausente.
- **HIERARQUIA DE FOCO:** Botoes de ajuste (Audio/Temas) agora possuem IDs e labels ARIA explicitos para facilitar o uso por tecnologias assistivas.
